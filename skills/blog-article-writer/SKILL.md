# Blog Article Writer スキル

BLOG_RULESの品質3原則とトーンガイドに準拠したブログ記事を作成するスキルです。

## 背景：なぜこのスキルが必要か

過去に以下の問題で記事の **全面リライト** が必要になりました：

- 相関関係を因果関係として記述（論理的矛盾）
- 同じ主張を異なる表現で複数箇所に分散（冗長性）
- 出典のない統計データの引用（事実ベースの欠如）
- 上から目線のトーン（CxO読者への配慮不足）
- 「まとめ」が本文の繰り返し

全面リライトは最もコストの高い失敗です。このスキルで初稿の品質を保証します。

## コンテンツ品質3原則

### 原則1: 論理的矛盾の排除

- 統計データが主張を **直接** 支えているか検証する
- 相関関係を因果関係として書かない
  - NG: 「DX投資額が増加しており、業績向上につながっている」
  - OK: 「DX投資額と業績向上に正の相関が報告されている（出典）が、因果関係は個別に検証が必要」
- 1つの事例で複数の提言すべてを証明しているように見せない
- 根拠が弱い箇所は「〜とされている」「〜の傾向がある」で区別する

### 原則2: 重複・冗長性の除外

- 1つの論点は1箇所で完結させる
- 「まとめ」セクションで本文をそのまま繰り返さない → **新しい視点や行動提案** を加える
- 冗長な修飾語を削る（「言うまでもなく」「ご存知の通り」等）
- 記事全体を通読し、同じメッセージが複数箇所に分散していないか確認する

### 原則3: 事実に基づく記述

- 統計・調査データには **出典（調査名・発行元・年・対象規模）** を明記
  - OK: 「IPA『情報セキュリティ10大脅威 2025』によると」
  - NG: 「調査によると約7割が…」
- 参考文献セクションに12〜16件の外部リンクを掲載
- 推測・伝聞は「〜とされている」「〜と指摘されている」で区別

## トーンガイド

### 基本スタンス
**「教える」ではなく「一緒に考える」**

### 具体的なルール

| ルール | OK例 | NG例 |
|--------|------|------|
| 共感から始める | 「〜に悩んでいる方も多いのではないでしょうか」 | 「〜を知らないのは問題です」 |
| 断定を避ける | 「〜が効果的とされています」 | 「〜すべきです」 |
| CxO配慮 | 「判断に迷うのは自然なことです」 | 「対策を怠っている企業は…」 |
| 安心に導く | 「段階的に取り組むことで改善できます」 | 「今すぐ対応しないと危険です」 |
| 専門用語 | 「SIEM（セキュリティ情報の一元管理ツール）」 | 「SIEMを導入してください」 |

### 禁止表現
- 上から目線の断定（「〜すべき」「〜しなければならない」の多用）
- 恐怖訴求（「被害に遭います」「手遅れになります」）
- 読者の過去の判断を否定する表現

## 記事構造テンプレート

```
H1: タイトル（40〜60文字、SEOキーワード含む）
├── メタ情報（日付 YYYY.MM.DD + カテゴリバッジ）
├── サマリボックス（「1分でわかるサマリ」4〜5箇条書き）
│
├── H2: セクション1（導入・問題提起）
│   ├── H3: 読者の悩みへの共感
│   └── H3: 記事で得られること
│
├── H2: セクション2〜6（本論）
│   ├── H3: データ・調査結果（表・統計を含む）
│   ├── H3: 事例（実企業名 — KADOKAWA、アスクル等）
│   └── H3: 具体的な対策・提言
│
├── H2: まとめ（本文の繰り返しNG → 新しい視点・行動提案）
│
├── 参考文献セクション（12〜16件、外部リンク）
│   - 形式: [組織名]「タイトル」(YYYY年)
│
├── FAQセクション（4〜5件）
│   - JSON-LD FAQPage スキーマ対応
│   - 実際に検索されそうな質問を選ぶ
│
├── CTAボックス
│   - 「まずは現状診断から始めませんか？」等
│   - サービスページへのリンク
│
└── 「記事一覧に戻る」リンク → /blog/
```

## ワークフロー

### フェーズ1: 企画

1. ユーザーからテーマ・キーワードを受け取る
2. ターゲット読者を確認（CxO向け / IT担当者向け）
3. slug（kebab-case英語）を決定
4. 記事構成（H2×5-8）のアウトラインを提案

### フェーズ2: 執筆

1. アウトライン承認後、セクションごとに執筆
2. 各セクションで品質3原則を適用
3. 参考文献は執筆と並行してリスト化

### フェーズ3: セルフレビュー

執筆完了後、以下のチェックリストで自己レビューを実施：

- [ ] **論理チェック**: 統計→主張の因果関係は正しいか？
- [ ] **冗長チェック**: 同じメッセージが複数箇所にないか？
- [ ] **出典チェック**: すべてのデータに調査名・年・規模があるか？
- [ ] **トーンチェック**: 上から目線の表現がないか？
- [ ] **構造チェック**: サマリ・FAQ・CTA・参考文献は揃っているか？
- [ ] **まとめチェック**: 「まとめ」に新しい視点があるか？
- [ ] **文字数チェック**: og:description が100文字以上か？

### フェーズ4: HTML生成

1. 記事HTMLを生成（headテンプレート、OGP、JSON-LD込み）
2. インラインCSSをブログ記事テンプレートに準拠して記述
3. レスポンシブ対応（clamp()関数、768pxブレークポイント）

## HTML生成時の必須要素

### head内（参照: references/head-template.md に詳細テンプレートあり）
- charset, viewport, author, canonical
- GA4 トラッキング（G-55QTMHWXX0）
- CSP meta タグ
- Google Fonts preconnect
- OGP タグ一式（og:description 100文字以上）
- Twitter Card タグ一式
- JSON-LD `@graph`（Article + FAQPage + BreadcrumbList）

### body
- Toast通知 (#toast-popup)
- ナビゲーション（全ページ共通パターン）
- `.page-hero` セクション
- `.blog-article` メインコンテンツ
- フッター（全ページ共通パターン）
- 共通JS読み込み（main.js, blog-data.js）

## ブランドガイド

| 要素 | 値 |
|------|-----|
| メインカラー | `#6B8F71` (--g1) |
| テキスト色 | `#3A4A3C` (--text) |
| 背景色 | `#F7F5F0` (--bg) |
| 日本語フォント | Noto Sans JP |
| 英数字フォント | Plus Jakarta Sans |
| 日付フォーマット（表示） | `YYYY.MM.DD` |
| 日付フォーマット（HTML属性） | `YYYY-MM-DD` |

## エラーハンドリング

| 問題 | 対処 |
|------|------|
| テーマが広すぎる | スコープを絞る提案（例：「セキュリティ全般」→「中小企業のCSIRT構築」） |
| 統計データが見つからない | 代替の出典を提案、またはデータなしの記述パターンに切り替え |
| 記事が長すぎる | H2を減らし、各セクションの文字数を制限 |
| トーンが合わない | レビューフェーズで具体的な修正箇所を指摘 |

## 注意事項

- このスキルは記事の **作成** に特化。公開登録は `blog-publisher` スキルで行う
- 記事HTMLの **検証** は `html-validator` スキルで行う
- 既存記事の **レビュー・改善提案** にもこのスキルを使用できる
